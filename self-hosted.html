Deployment paths

One gate contract, three ways to run it.

01 Open source

Run it yourself.

Developers can inspect the basic package, run deterministic examples, and integrate it into an existing chatbot or application without a hosted account.

02 Self-hosted enterprise

Keep sensitive data inside.

Banks, medical teams, governments, studios, and large enterprises can deploy the gate in their own VPC, data center, or private cloud. SemeAI does not need to see their documents.

03 SaaS API

Start quickly when policy allows.

Smaller teams and pilots can use the hosted API for fast evaluation. The API returns SHOW, REVIEW, or BLOCK plus audit metadata.

Security answer

If documents cannot leave, they should not leave.

The correct answer to a CTO or security team is not "trust our SaaS anyway." The correct answer is: run SemeAI Gate inside your boundary and keep your documents, prompts, business rules, receipts, and review queues under your own control.

SaaS is for quick pilots and lower-sensitivity workflows. Self-hosted is for teams whose policies do not permit sensitive material to leave their infrastructure.

What this does not claim

Conservative by design.

Not universal hallucination detection.

The gate checks scoped business evidence, rules, context, and release-risk patterns.

Not a model replacement.

Existing LLMs still generate candidate answers. SemeAI decides whether they should be released.

Not compliance certification.

Self-hosted deployment helps with data-boundary requirements, but formal compliance still belongs to the operator.
